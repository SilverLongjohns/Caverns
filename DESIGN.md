# Caverns — Game Design Document

## Overview

**Caverns** is a cooperative text-based dungeon crawler for 1–4 players, hosted as a web application. Players explore procedurally generated cave systems in real time, fight monsters in turn-based combat, collect loot, and attempt to reach and defeat the boss on the final floor. Each run is a self-contained roguelike session — no persistent progression between runs.

### Tech Stack

- **Frontend:** React / TypeScript
- **Backend:** Node.js / TypeScript (WebSocket server for real-time state)
- **Content Generation:** Claude API (pre-generates session content as JSON at run start)

### Core Design Pillars

- **Emergent roles:** No classes. Your role in the party is defined by the gear you find.
- **Risk vs. reward:** Splitting up covers more ground but risks solo encounters with dangerous mobs.
- **Procedural variety:** Each run has a unique cavern with themed rooms, enemies, loot, and flavor text generated by Claude at session start.
- **Simple mechanics, interesting decisions:** Movement is cardinal directions on a room graph. Combat is turn-based. Complexity comes from party coordination and gear choices.

---

## Run Flow

1. **Lobby** — 1–4 players form a party.
2. **Generation** — Server calls Claude API to generate the session content (rooms, mobs, loot, events, boss) as a themed JSON blob.
3. **Enter Cavern** — All players start in the entrance room of Floor 1 with starter gear.
4. **Explore** — Players move between rooms in real time via cardinal directions (N/S/E/W). Fog of war is shared — any player revealing a room reveals it for all.
5. **Encounter** — Entering a room with mobs initiates turn-based combat for players in that room. Other players can move to the room to join mid-combat.
6. **Descend** — Finding the staircase down advances to the next floor. All party members must be at the staircase to descend (keeps the group loosely together).
7. **Boss** — The final floor contains a boss encounter (party-required).
8. **Victory or Wipe** — Kill the boss to win. If all players are downed simultaneously, the run ends in a wipe.

## Map & Exploration

### Structure

The cavern is a **node graph** (not a strict 2D grid). Each node is a room connected to other rooms via cardinal exits. This is the classic MUD model — rooms don't need spatial consistency on a coordinate plane, just logical connections.

Each run has **3–5 floors** (tunable as a difficulty setting). Each floor is a self-contained cluster of rooms with:
- One entry point (staircase from above, or the cavern entrance on Floor 1)
- One exit point (staircase down, or the boss room on the final floor)
- 10–20 rooms per floor (scales with floor number)

### Room Types

| Type | Description | Contents |
|------|-------------|----------|
| **Tunnel** | Narrow connecting passages | Mostly empty, occasional minor loot or ambient events |
| **Chamber** | Standard rooms | Mob encounters, loot chests, interactive objects |
| **Cavern** | Large open areas | Elite/multi-skull encounters, significant loot |
| **Dead End** | Risk/reward rooms | Could be a treasure cache, a trap, or an ambush |
| **Staircase** | Floor transition | Gathering point for descent; may have a guardian encounter |

### Movement

- Players move independently in real time by choosing a cardinal direction (N, S, E, W).
- Movement is instant (no travel time between connected rooms).
- Players can see which rooms are explored, which players are in which room, and the exit directions of explored rooms.
- **Fog of war** is shared across the party — if one player scouts ahead, everyone's map updates.

### Splitting Up

Players are free to explore independently. This is strategically useful for scouting but risky:
- A solo player who stumbles into a multi-skull room is in trouble.
- A downed player can only be revived by a party member in the same room.
- Splitting up is a calculated risk, not a failure state.

## Combat System

### Initiation

When a player enters a room containing hostile mobs, combat begins for all players currently in that room. Combat is **turn-based** — the real-time exploration pauses for participants.

- Players **not** in the room continue exploring in real time.
- A player who moves into an active combat room **joins the combat** at the start of the next round.
- Players can choose to **flee** on their turn (exits combat and moves to an adjacent room).

### Turn Order

Initiative is determined per round:
1. All participants (players + enemies) are ordered by an initiative stat (base value + gear modifiers + small random factor).
2. Each participant takes one action per turn.
3. After all participants act, a new round begins.

### Actions

On their turn, a player can:
- **Attack** — Use equipped weapon against a target. Damage = weapon damage + modifiers.
- **Defend** — Reduce incoming damage until next turn. Useful for tanks drawing aggro.
- **Use Item** — Consume a potion, throwable, or scroll from the consumable pouch.
- **Ability** — Some gear grants active abilities (e.g., a staff that can heal an ally, a shield that taunts all enemies).
- **Flee** — Exit combat and move to an adjacent room. Enemies may get an opportunity attack.

### Threat Rating (Skull System)

Every mob/encounter has a visible **skull rating** that indicates difficulty:

| Rating | Meaning | Design Intent |
|--------|---------|---------------|
| ☠ | Solo-capable | Any single player with reasonable gear can handle this. Common trash mobs. |
| ☠☠ | Pair recommended | Challenging solo. High HP or damage. Comfortable with 2 players. |
| ☠☠☠ | Party recommended | Dangerous mechanics that demand role coverage (tanking, healing, burst DPS). |

Skull ratings are visible **before engaging**, allowing tactical decisions (fight solo, call for backup, or skip and come back later).

### Multi-Skull Mob Mechanics

Higher-skull enemies should require party coordination through specific traits:
- **High sustained damage** — Needs a tanky player to absorb hits while others deal damage.
- **Damage resistance / armor** — Needs multiple damage sources or armor-piercing abilities.
- **Healing / summoning** — Needs burst DPS from multiple players to overcome regeneration or add spawns.
- **Enrage timers** — Gets stronger over time; rewards fast coordinated damage.

### Defeat & Revival

- A player at 0 HP is **downed** (incapacitated, not dead).
- A downed player can be revived by another player in the same room using a turn action or a consumable.
- If **all players are downed simultaneously**, the run ends in a **wipe** — game over.
- A solo player going down in a room alone must wait for a party member to reach them.

## Loot System

### Rarity Tiers

| Tier | Color | Drop Weight | Notes |
|------|-------|-------------|-------|
| **Common** | White/Gray | High | Basic stat boosts. Functional but unexciting. |
| **Uncommon** | Green | Moderate | Noticeable upgrades. May have a minor secondary effect. |
| **Rare** | Blue | Low | Strong items with meaningful special abilities or high stats. |
| **Legendary** | Gold/Orange | Very Low | Build-defining items. Unique abilities that shape playstyle. |

Drop rates shift with depth — deeper floors weight toward higher rarities.

### Equipment Slots

Each player has **4 equipment slots**:

| Slot | Examples | Role Impact |
|------|----------|-------------|
| **Weapon** | Sword, dagger, staff, bow, hammer | Primary damage source. Determines attack style and some abilities. |
| **Off-hand** | Shield, torch, orb, second dagger | Defines defensive/utility role. Shield = tank. Orb = support. Torch = scouting. |
| **Armor** | Leather, chainmail, robes, plate | Survivability. Heavy armor = tank. Robes = might grant support abilities. |
| **Accessory** | Ring, amulet, charm, belt | Specialized bonuses. Crit chance, healing power, movement-related perks. |

### Consumables

Separate from equipment — a **consumable pouch** with 4–6 slots:
- **Health potions** — Restore HP.
- **Throwables** — Deal damage or apply debuffs (acid flask, smoke bomb).
- **Scrolls** — One-use abilities (AoE damage, party heal, reveal map).
- **Food/buffs** — Temporary stat boosts.

### Starter Loadout

Every player begins a run with:
- A basic weapon (e.g., "Rusty Shortsword" — low damage, no special properties)
- No armor, no off-hand, no accessory
- 2 minor health potions

This ensures gear found during the run feels immediately impactful.

### Loot Distribution (Co-op)

When loot drops (from mobs, chests, or caches), it is **visible to all players in the room**. Distribution uses a **need/greed** prompt:

- **Need** — "I want this for my build." Priority roll.
- **Greed** — "I'll take it if nobody needs it." Secondary roll.
- **Pass** — "I don't want it."

Ties are resolved by random roll. Quick timer (10–15 seconds) to keep the pace up. Solo players auto-receive all drops.

## Gear-Driven Roles

Since there are no classes, a player's role emerges entirely from equipped gear. The system should make it so that gear combinations naturally push toward recognizable archetypes:

### Tank
- Shield (off-hand) + heavy armor + HP/defense accessories
- Gear abilities: Taunt (force enemies to target you), Block (reduce incoming damage), Fortify (temporary HP)

### DPS
- High-damage weapon + damage-boosting accessory + light armor for speed
- Gear abilities: Critical Strike, Multi-hit, Armor Pierce, Execute (bonus damage to low-HP enemies)

### Support / Healer
- Staff or totem (weapon) + orb (off-hand) + robes + healing accessory
- Gear abilities: Heal Ally, Party Shield, Cleanse Debuff, Buff Attack

### Utility / Scout
- Dagger or ranged weapon + torch (off-hand) + light armor + scouting accessory
- Gear abilities: Reveal Rooms, Disarm Trap, Sneak (avoid combat), Loot Bonus

These are emergent tendencies, not rigid classes. A player might end up with a shield and a healing staff — a tanky healer hybrid. The fun is in adapting to what you find.

## Content Generation (Claude Integration)

### Session Generation

At the start of each run, the server sends a prompt to the Claude API requesting a complete session content package. The prompt includes:
- Party size
- Target floor count
- Difficulty setting (if applicable)
- A seed/theme hint (or let Claude pick)

### Generation Strategy

- **One API call per run** — batch-generate all content upfront. No mid-game API calls (avoids latency during play).
- **Themed consistency** — the prompt asks Claude to maintain thematic coherence (enemy types, loot flavor, room descriptions all match the cavern theme).
- **Stat guardrails** — the prompt provides stat ranges and balance constraints so generated content is mechanically sound (e.g., "Floor 1 mobs should have 20–40 HP and deal 5–12 damage").
- **Fallback content** — a set of hand-authored generic rooms, mobs, and loot exists as a fallback if the API call fails.

## Networking & Architecture

### Server

- **Node.js / TypeScript** WebSocket server.
- Manages authoritative game state: player positions, inventories, room states, combat encounters.
- Receives player actions (move, attack, use item, loot choice) and broadcasts state updates.
- Handles the Claude API call for session generation at run start.

### Client

- **React / TypeScript** frontend.
- Renders the map (explored rooms, player positions, fog of war).
- Renders the text log (room descriptions, combat narration, event outcomes).
- Sends player actions via WebSocket.
- Receives state updates and re-renders.

### State Management

- Server is the **single source of truth** for all game state.
- Client maintains a local view of the state for rendering.
- All player actions are validated server-side (no client authority).

## UI Concept

The interface is primarily **text-based with a minimal map overlay**:

### Main Panels

1. **Text Log** — The primary panel. Scrolling text output showing room descriptions, combat play-by-play, loot drops, event narratives, party chat. This is the heart of the game.
2. **Map** — A small node-graph minimap showing explored rooms, player positions (color-coded), and exits. Rooms are simple shapes connected by lines — not a graphical dungeon renderer.
3. **Player HUD** — Current HP, equipped gear summary, consumable pouch, status effects.
4. **Party Panel** — Other players' names, HP bars, current room, status (exploring / in combat / downed).
5. **Action Bar** — Context-sensitive. In exploration: directional movement buttons + interaction. In combat: attack / defend / item / ability / flee.

### Text Log Style

The text log should read like a tabletop narration:

```
--- Fungal Grotto ---
A low-ceilinged chamber carpeted in luminous mushrooms. Water pools in the center, reflecting the eerie glow.

Exits: [North] [East]

A Fungal Crawler skitters toward you from behind a mushroom cluster! (☠)

> [Fight] [Flee North] [Flee East]
```

## Future Considerations (Out of Scope for V1)

These are explicitly **not** part of the initial build but are noted for future expansion:

- **Meta-progression** — Persistent unlocks between runs (new starter gear, cosmetic titles, unlockable cavern themes).
- **Class system** — Optional starting classes that provide a baseline kit, with gear still modifying role.
- **Extraction mode** — Alternative to completion: extract with your loot at any time via the entrance, but going deeper yields better rewards.
- **PvP / competitive** — Rival parties in the same cavern competing for loot.
- **Real-time combat** — Hybrid system where basic attacks are real-time but abilities are turn-gated.
- **Persistent world** — A hub town between runs with shops, NPC quests, party management.
- **Claude narrator mode** — Real-time Claude API calls that narrate combat outcomes and player actions with dynamic flavor text based on the current game state.
